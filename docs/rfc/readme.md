# RFC (Request for Comments)

RFC는 "의견을 요청하는 문서"라는 뜻으로, 새로운 기능 도입, 시스템 설계 변경, 정책 결정 등을 하기 전에 팀원들에게 자신의 아이디어를
제안하고 피드백을 수집하기 위해 작성하는 문서입니다.

## RFC란 무엇인가?

RFC는 소프트웨어 공학에서 널리 쓰이는 협업 방식 중 하나입니다. 결정이 내려지기 전에 이해관계자들과 소통하며 잠재적인 문제를 조기에
발견하고, 합의를 이끌어내는 데 목적이 있습니다.

- **투명성**: 결정 과정을 공개하여 누구나 맥락을 이해할 수 있게 합니다.
- **기록**: 왜 그런 결정을 내렸는지 히스토리를 남깁니다.
- **품질 향상**: 다양한 시각의 피드백을 통해 설계의 빈틈을 메웁니다.

## RFC 작성 단계

1. **초안 작성**: 제안하고자 하는 아이디어를 문서화합니다.
2. **공유 및 피드백**: 팀 내에 공유하고 일정 기간 의견을 받습니다.
3. **수정 및 보완**: 피드백을 반영하여 내용을 다듬습니다.
4. **결정**: 승인(Accepted), 거절(Rejected), 또는 보류(Deferred) 상태로 변경합니다.

## RFC 문서 포함 내용 (템플릿 제안)

### 1. 요약 (Summary)

제안의 핵심 내용을 한두 문장으로 요약합니다.

### 2. 배경 및 동기 (Motivation)

왜 이 변경이 필요한지, 현재 어떤 문제가 있는지 설명합니다.

### 3. 상세 설계 (Detailed Design)

구체적으로 어떻게 구현할 것인지 설명합니다. 아키텍처 다이어그램, API 명세, 코드 예시 등을 포함할 수 있습니다.

### 4. 장단점 (Pros and Cons)

이 제안을 도입했을 때의 이득과 감수해야 할 비용(복잡도, 성능 저하 등)을 나열합니다.

### 5. 대안 (Alternatives)

고려했던 다른 방법들과 왜 그 방법 대신 이 제안을 선택했는지 설명합니다.

### 6. 질문 및 논의 사항 (Unresolved Questions)

아직 확정되지 않았거나 팀원들의 집중적인 의견이 필요한 부분을 명시합니다.

## 규칙

- RFC는 `docs/rfc/` 디렉토리에 `YYYYMMDD-subject.md` 형식으로 작성합니다.
- 작성 후 슬랙이나 협업 툴을 통해 팀에 공유합니다.
